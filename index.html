The Philosophy

Design thinking
meets the new tools.

I've spent a lifetime making things look right, work right, and mean something. Then AI changed what "building software" could mean — and I paid attention.

My background is in graphic design, document management, and project management. That's a career built on systems thinking, visual judgment, and knowing how to get a complex thing across a finish line. I've also been writing code for most of that time — PHP, HTML, CSS, and a handful of other languages — so I know what good software looks like from the inside.

A conductor doesn't play every instrument. They know the score, they hear what's off, and they know exactly how to get what they want from the ensemble.

That's how I build now. I could write this code myself — I have, many times — but AI does it faster and with fewer mistakes than I ever could alone. So I direct: precise instructions, informed by design principles and domain knowledge, with a clear picture of what good looks like. The AI handles execution. I decide if it's right.

The projects on this site are the proof of concept — production-grade software built the way I think more software should be built. Taste and judgment aren't soft skills. They're the whole job.

The Method

The Madness
of a Study.

The idea was simple: a comfortable place on the internet that feels like home. Not a portfolio in the traditional sense. Not a resumé with scroll effects. Something more like a study — a room where work accumulates thoughtfully, where you can tell a real person lives here.

From that room, other rooms branch off. Each project lives in its own space — its own subdomain, its own personality — but the same person is clearly at home in all of them. The Dad-a-Base is the first. More are on the way. The study is the hub. The work is the point.

I wanted to make something that didn't try to look like everyone else's site, didn't hedge, and wasn't afraid to be a little bit itself. Real work, from a real person, built with real care. The method behind it — thirty years of design thinking combined with new tools used deliberately — is what the rest of this section is about.

The Approach

The work happens in conversation with Claude — directed precisely, reviewed carefully, and iterated until it's right. That's not a workaround for anything. It's a different application of the same skills that make any complex creative project work: knowing what good looks like, being able to describe it clearly, and recognizing when you haven't got there yet.

The coding background helps. PHP, HTML, CSS — years of it. Enough to read what comes back, catch something structurally off, and ask the right follow-up. Understanding the craft you're directing is part of directing it well.

The Workflow

Each working session begins with context-setting: the README goes in first, establishing the project state. Then I describe what I want — not in vague terms, but with specificity about layout, behavior, copy, and edge cases. The more precisely I can describe good, the less time we spend on wrong.

Claude writes the code. I read it — not line by line, but well enough to catch something structurally off. I review the output in a browser. I note what's right and what isn't. Then we iterate. Most features take two or three passes. Some take more. A few come out nearly right the first time.

When the session ends, the README gets updated. Decisions made, copy settled, open questions resolved or added. The document is the memory. Next session, we start clean and still know everything.

The Tools

The primary tool is Claude (Anthropic) — specifically Claude's computer use and artifact capabilities, which allow working directly with files in a live environment rather than copying and pasting code back and forth. That changes the texture of the work significantly. It feels less like prompting and more like pairing.

Everything else in the stack is deliberately minimal: PHP, HTML, CSS, vanilla JavaScript. No frameworks, no build step, no npm. Partly because the hosting environment is shared hosting and simplicity is a virtue. Partly because complexity you don't need is just places for things to break.

What Goes Wrong (And How It Gets Fixed)

The process works well. It also fails in predictable ways. Three patterns come up often enough to be worth naming:

Context Drift
In a long session, earlier decisions can quietly get contradicted by later ones — especially on copy and CSS. The README is the fix: decisions written down don't drift. Anything not in the document is an open question, whether it feels settled or not.
Confident Wrong Answers
AI is fluent in being wrong. The output looks right, reads right, and quietly doesn't work — or works in the browser I tested but breaks somewhere else. The defense is knowing enough to be suspicious, and reviewing outputs rather than just trusting them. Experience is what tells you which answers to double-check.
Scope Creep in Both Directions
Sometimes the AI adds features I didn't ask for and don't want. Sometimes I ask for something vague and get something technically correct but completely wrong in spirit. Precision in the brief is the only fix — which, it turns out, is also the fix for scope creep in every other kind of project.
Why This Matters

The question I get — implicitly, in every conversation about this work — is whether it counts. Whether software built this way is real.

The answer is yes, and the evidence is what ships. The Dad-a-Base is a fully functional web application with a database, a moderation system, live voting, and an admin panel. It runs on a real server. Real people use it. The method doesn't change what the thing is.

A contractor who manages a build isn't less responsible for the house because they didn't swing every hammer. The judgment, the taste, the decisions about what to build and how it should work — that's the job. Execution is a detail.
